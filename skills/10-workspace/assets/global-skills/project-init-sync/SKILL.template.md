# 專案初始化同步

設定專案工作模式，但不要製作第一個班級工具。

## 確認資料

確認專案名稱、用途、資料夾、是否需要 GitHub、儲存庫公開或私人、是否需要 GitHub Pages、Obsidian 駕駛艙位置、是否使用 Firebase，以及部署目標。只詢問無法安全查得的資料。GitHub 帳號未另行指定時，預設使用 `changyiwu`。

## 執行流程

1. 檢查目標資料夾，回報已存在與缺少的項目。
2. 保留既有的 `AGENTS.md`、`README.md`、`.gitignore`、Git 歷史、遠端設定與 Firebase 設定。只加入相容的內容。
3. 建立或補齊：
   - `AGENTS.md`：記錄專案入口、路徑、Obsidian 連結、Firebase 專案、同步規則、主要檔案與安全邊界。
   - `README.md`：記錄用途與適合初學者的使用方式。
   - `.gitignore`：排除系統檔、敏感資料、代理程式本機設定、相依套件與建置輸出。
   - Obsidian 駕駛艙：固定放在設定好的 Vault 根目錄，檔名使用 `<專案名稱>-專案駕駛艙.md`；不要建立駕駛艙子資料夾。記錄最後動作、狀態、下一步、工具清單、變更紀錄與踩坑筆記。若同名筆記已存在，先讀取並合併，不要覆寫。建立、補齊、移動或重新命名駕駛艙時，不要更新 `知識庫/log.md` 或其他全域操作紀錄。
4. 只有在尚未使用 Git 時才初始化。若儲存庫位於雲端同步資料夾，設定 `windows.appendAtomically=false`。
5. 只有在使用者明確要求時，才建立或連接 GitHub 儲存庫。建立前確認公開或私人，以及是否需要 GitHub Pages。
   - 可寫入的 `origin` 預設設為 `https://github.com/changyiwu/<repository>.git`。
   - 若專案來源屬於其他帳號，將原作者儲存庫保留為 `upstream`，不要當成預設推送目的地。
   - 新儲存庫與本機工作分支預設使用 `main`；若既有儲存庫的預設分支不同，先回報差異再調整。
6. 使用者明確要求 commit 或 push 時，預設直接提交並推送到 `origin/main`，不要建立功能分支或 Pull Request；只有使用者當次另有指定時才使用分支或 PR。
7. 推送前先抓取並確認 `origin/main`、提交範圍與工作區狀態。不要 force push；若有未預期變更、衝突或分支保護，停止並回報。
8. 初始化工作區時不要部署，也不要修改 Firebase 安全規則。
9. 驗證路徑、檔案、Git 狀態、遠端網址與 Obsidian 筆記是否可讀。

## 完成交接

回報已建立與保留的檔案、GitHub 網址（若有）、Obsidian 駕駛艙路徑與下一步。提醒使用者：「開工」會載入情境，「收工」會準備安全交接。

## 安全規則

- 不要覆蓋既有專案規則或無關變更。
- 不要把專案駕駛艙建立在工作資料夾或 Vault 子資料夾；初始化狀態只寫在駕駛艙本身。
- 不要提交 `.env`、憑證、權杖、密碼、`.codex/` 或 `.claude/`。
- 學生紀錄只使用座號與班級代碼，不要儲存真實姓名。
- 將建立 GitHub、變更公開狀態、啟用 Pages、部署、提交與推送視為必須由使用者明確選擇的操作。
- 不要因一般發佈慣例自行建立分支或 PR；本工作流的預設是獲得推送授權後直接同步 `main`。
